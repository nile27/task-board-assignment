# 설계 결정 (DECISIONS)

> 이 문서는 평가에서 **20%**를 차지합니다. "무엇을 했는지"보다 **"왜 그렇게 했는지"**를 봅니다.

## 개요 - 나만의 원칙

이번 과제에서는 라이브러리 사용을 최소화하고 직접 구현하는 것을 목표로 삼았습니다.
특히 로딩·에러 처리는 TanStack Query의 `useQuery`가 제공하는 상태값(`isLoading`,
`isError`)으로, 낙관적 업데이트는 `useMutation`의 `onMutate`/`onError`/`onSuccess`
콜백으로 편리하게 구현할 수 있다는 것을 알고 있었지만, 이번 과제는 기본적인 React
훅을 어떻게 활용하는지를 보여주는 데 취지가 있다고 판단해 라이브러리 사용을
최소화했습니다. 다만 가상화는 처음 접해본 전략이었고, 기본 훅만으로 직접
구현하기에는 현재 제 실력으로 한계가 있다고 판단해, 이 부분에 한해
라이브러리(`@tanstack/react-virtual`)를 도입했습니다.

## 1. 상태 구조

- 서버 상태와 클라이언트(낙관적) 상태를 어떻게 나눴나요? 어디서 관리하나요?
  - 별도 분리 안 함. `tasks`(useState<Task[]>) 하나에서 서버값+낙관적값 함께 관리.
  - `inFlight`(useRef<Set>)로 "요청 중인 카드"만 별도 추적.

- 이렇게 잡은 이유와, 고려했다가 버린 대안은?
  - 이유: 모든 조작이 "낙관적 반영 → 요청 → 성공 시 교체 / 실패 시 롤백" 동일 패턴이라, 분리하면 동기화 코드만 늘어남.
  - 훅 선택: `useState`=화면 반영 필요한 값 / `useRef`=로직 판단용(리렌더 불필요) / `useEffect`=마운트 1회성 동작.
  - useRef 이유: 일반 변수는 리렌더마다 초기화됨. useState는 불필요한 리렌더 유발. useRef는 값 유지 + 리렌더 없음.
  - 버린 대안: TanStack Query — 편의는 크지만 과제 취지(기본 훅 활용)에 안 맞아 미채택.

## 2. 낙관적 업데이트 & 롤백

- 이동/수정/삭제/생성 시 UI를 먼저 반영하고 실패하면 어떻게 되돌리나요?
  - 이동: status 즉시 변경(patchTask) → 실패 시 이전 카드로 복원(rollbackTask)
  - 수정: 변경값 즉시 반영(patchTask) → 실패 시 이전 카드로 복원(rollbackTask)
  - 삭제: 목록에서 즉시 제거(removeById) → 실패 시 배열 끝에 재삽입
  - 생성: 임시 id로 즉시 추가 → 성공 시 진짜 카드로 교체, 실패 시 제거(removeById)

- 롤백을 "서버 재조회" vs "이전 스냅샷 복원" 중 무엇으로 했고 그 이유는?
  - 스냅샷 복원 선택.
  - 이유: 5,000개라 재조회 비용 큼(지연+재파싱), 낙관적 취지와도 안 맞음.
  - 전체가 아닌 해당 카드만 복원: 그 사이 다른 카드의 성공한 변경까지 같이 사라지는 것 방지.
  - *(처음엔 전체 스냅샷 복원이었다가 이 문제 발견 후 카드별 복원으로 수정함 — 아래 6번 한계에도 연결)*

## 3. 경쟁 상태(race condition)

- 같은 카드를 빠르게 연속 이동할 때 최종 상태가 어긋나지 않도록 무엇을 했나요?
  - 카드별 요청 직렬화. `inFlight`(Set)에 요청 중인 카드 id 저장, 요청 중이면 새 요청 무시, 끝나면(finally) 해제.
  - 효과: 한 카드당 항상 요청 1개만 진행 → 응답 순서 뒤섞임/옛 version 충돌(409) 방지.
  - 디바운스+클린업 방식(200ms 지연 후 전송, 재이동 시 clearTimeout으로 이전
    타이머 취소)도 직접 구현해봤으나 아래 문제로 채택하지 않음:
    - version 타이밍: 지연(200ms)이 서버 응답 시간(200~800ms)보다 짧아, 다음
      전송 시점에 직전 응답의 최신 version이 아직 반영 안 됐을 수 있어 409
      가능성이 남음.
    - **지연 시간 중 새로고침/탭 닫기 시 유실:** setTimeout으로 예약된 전송은
      브라우저 메모리에만 존재해, 200ms 대기 중에 새로고침하거나 탭을 닫으면
      예약된 요청 자체가 사라져 서버에 전혀 반영되지 않음. 즉시 전송하는
      직렬화는 이 유실 창을 최소화함.
    - 위 이유로 지연 전송(디바운스) 대신 즉시 전송 + 요청 중 차단(직렬화)으로
      최종 확정.
  
  - 검토 후 미채택 — abort: GET엔 적합하나 PATCH엔 부적합(서버는 이미 처리, 클라만 취소해 version 꼬임).

## 4. 대량 데이터 성능 (5,000개)

- 검색/필터/드래그를 부드럽게 만들기 위해 무엇을 선택했고 **무엇을 포기**했나요?
  - `@tanstack/react-virtual`로 컬럼별 리스트를 가상화(Column.tsx). 화면에 보이는
    카드 + overscan만 실제 DOM에 렌더.
  - 가변 높이 카드 대응을 위해 `measureElement`로 실제 높이 측정(예상 높이만
    쓰면 카드 간 간격이 어긋나는 문제가 있어 수정함).
  - 드래그/드롭은 카드 단위가 아니라 컬럼(`section`) 전체가 드롭 타깃이라,
    가상화로 화면 밖 카드가 DOM에서 사라져도 드롭 동작에 영향 없음.
  - 포기한 것: 카드 순서 유지/재배치는 다루지 않음(상태 변경만). 검색/필터
    기능 자체는 시간상 미구현.
  - 라이브러리 선택 이유(개요 참고): 스크롤 위치 계산, 가변 높이 처리는
    직접 구현하기엔 현재 실력으로 한계가 있다고 판단해 라이브러리 도입.

- 가능하면 측정 근거(예: 최적화 전/후 체감·프로파일)를 남겨 주세요.
  - 최적화 전: `document.querySelectorAll('.card').length`로 확인한 실제 렌더된
    카드 수가 5,000에 가까웠고, 스크롤 시 버벅임이 체감됨.
  - 최적화 후: 같은 방법으로 확인 시 40~60개 수준으로 감소, 스크롤이
    부드러워짐.

## 5. 정답이 없던 결정들 (명세서에 명시되지 않은 항목)

- **네트워크 완전 단절** 시 동작을 어떻게 정의했나요? 왜?
  - 별도 분기 없이 일반 쓰기 실패(500 등)와 동일하게 처리. 실패 시 롤백 +
    alert로 사용자에게 알림.
  - 이유: 클라이언트 입장에서 "완전 단절"과 "서버 오류"를 구분할 근거가 없어,
    구분해도 사용자에게 해줄 수 있는 대응이 같음(다시 시도 유도).

- **409 충돌** 시 사용자에게 무엇을 보여주고 어떻게 해소하나요? 왜?
  - 롤백이 아니라 화해로 처리. 서버가 준 최신 상태(`payload.current`)로 해당
    카드를 교체하고 "다른 곳에서 먼저 변경되었습니다. 최신 상태로 맞췄어요"
    alert 표시.
  - 이유: 409는 내 요청이 틀린 게 아니라 그 사이 서버가 먼저 바뀐 것이므로,
    내 낙관적 변경을 버리는 것보다 서버 최신값을 반영하는 게 더 정확함.
    전체 재조회 대신 payload.current만 쓰는 이유는 5,000개 재조회 비용 회피.

- 실패한 쓰기의 **재시도**: 자동 vs 수동 — 무엇을 택했고 이유는?
  - 수동(사용자가 다시 시도). 실패 시 롤백 + alert만 하고 자동 재전송은 하지 않음.
  - 이유: 자동 재시도를 디바운스 실험 중 `fetchWithRetry`(최대 3회, 500ms 간격)로
    시도해봤으나, 이미 사용자가 다른 조작을 한 뒤 뒤늦게 재시도가 반영되면
    오히려 혼란을 줄 수 있다고 판단해 최종 구현에서는 제외. 초기 로드 실패에
    한해서는 "다시 시도" 버튼으로 수동 재시도 제공.
