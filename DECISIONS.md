## 개요 - 나만의 원칙
 
이번 과제에서는 라이브러리 사용을 최소화하고 직접 구현하는 것을 목표로 삼았습니다.
특히 로딩·에러 처리는 TanStack Query의 `useQuery`가 제공하는 상태값(`isLoading`,
`isError`)으로, 낙관적 업데이트는 `useMutation`의 `onMutate`/`onError`/`onSuccess`
콜백으로 편리하게 구현할 수 있다는 것을 알고 있었지만, 이번 과제는 기본적인 React
훅을 어떻게 활용하는지를 보여주는 데 취지가 있다고 판단해 라이브러리 사용을
최소화했습니다. 다만 가상화는 처음 접해본 전략이었고, 기본 훅만으로 직접
구현하기에는 현재 제 실력으로 한계가 있다고 판단해, 이 부분에 한해
라이브러리(`@tanstack/react-virtual`)를 도입했습니다.
 
## 1. 상태 구조
 
- 서버 상태와 클라이언트(낙관적) 상태를 어떻게 나눴나요? 어디서 관리하나요?
  - 별도로 분리하지 않았습니다. `Board.tsx`의 `tasks`(`useState<Task[]>`) 배열 하나에서
    서버 확정값과 낙관적 반영값을 함께 관리합니다. 다만 "지금 이 카드가 서버 확정을
    기다리는 중인가"만 `inFlight`(`useRef<Set<string>>`)로 별도 추적합니다.
- 이렇게 잡은 이유와, 고려했다가 버린 대안은?
  - **이유:** 이동/생성/수정/삭제가 전부 "낙관적 반영 → 요청 → 성공 시 교체 / 실패 시
    롤백"이라는 동일한 패턴입니다. 상태를 서버/클라이언트로 분리하면 오히려 두 상태를
    동기화하는 코드가 추가로 필요해져, 단일 배열로 관리하는 편이 더 단순하다고
    판단했습니다.
  - **훅 선택 기준:**
    - `useState`: 값이 바뀔 때 화면이 다시 그려져야 하는 것 (tasks, loading, isError,
      showForm, editing)
    - `useRef`: 로직 판단에만 쓰이고 화면과 무관한 것 (inFlight — 요청 중 여부)
    - `useEffect`: 마운트 시 1회성 동작 (초기 로드)
  - **일반 변수 대신 useRef를 쓴 이유:** 일반 변수(let)는 리렌더 시 컴포넌트 함수가
    처음부터 재실행되어 매번 초기화되므로 "요청 중" 정보가 유지되지 않습니다.
    `useState`는 값은 유지되지만 변경 시 불필요한 리렌더를 유발합니다(5,000개 카드가
    리렌더될 수 있음). `useRef`는 리렌더 간 값이 유지되면서 리렌더를 일으키지 않아
    이 용도에 적합했습니다.
  - **버린 대안 — TanStack Query:** 편의는 크지만, 과제 취지에 맞게 React가 제공하는
    훅들을 최대한 활용해 직접 구현하기 위해 채택하지 않았습니다.
## 2. 낙관적 업데이트 & 롤백
 
- 이동/수정/삭제/생성 시 UI를 먼저 반영하고 실패하면 어떻게 되돌리나요?
  - 네 가지 조작 모두 "화면 먼저 반영 → 서버 요청 → 성공 시 서버값 반영 / 실패 시
    되돌림" 흐름입니다.
    - **이동(moveTask):** status를 즉시 변경(`patchTask`) → 실패 시 이동 전 카드로
      복원(`rollbackTask`)
    - **수정(editTask):** 변경값을 즉시 반영(`patchTask`) → 실패 시 수정 전 카드로
      복원(`rollbackTask`)
    - **삭제(removeTask):** 목록에서 즉시 제거(`removeById`) → 실패 시 배열 끝에
      재삽입
    - **생성(addTask):** 임시 id(`temp-${Date.now()}`)로 카드를 즉시 추가 → 성공 시
      서버가 준 진짜 카드로 교체, 실패 시 임시 카드 제거(`removeById`) 
      
- 롤백을 "서버 재조회" vs "이전 스냅샷 복원" 중 무엇으로 했고 그 이유는?
  - **이전 스냅샷 복원**을 선택했습니다. 데이터가 5,000개라 실패 하나마다
    `getTasks()`로 전체를 다시 받아오면 비용이 크고(200~800ms 지연 + 5,000개
    재파싱), 그 사이 화면이 다시 로딩 상태로 보이거나 멈춰 보일 수 있어 낙관적
    업데이트의 취지와도 맞지 않습니다. 실패 이전에 붙잡아둔 해당 카드만 되돌리면
    즉시 복원되고 비용도 적습니다.
  - 전체 배열을 통째로 스냅샷 복원하지 않고 **해당 카드만** 되돌리는 이유는, 그
    사이 다른 카드에 대해 성공적으로 반영된 변경이 함께 사라지는 것을 막기
    위해서입니다.
