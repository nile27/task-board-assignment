# 설계 결정 (DECISIONS)
 
> 이 문서는 평가에서 **20%**를 차지합니다. "무엇을 했는지"보다 **"왜 그렇게 했는지"**를 봅니다.
 
## 개요 - 나만의 원칙
 
이번 과제에서는 라이브러리 사용을 최소화하고 직접 구현하는 것을 목표로 삼았습니다.
특히 로딩·에러 처리는 TanStack Query의 `useQuery`가 제공하는 상태값(`isLoading`,
`isError`)으로, 낙관적 업데이트는 `useMutation`의 `onMutate`/`onError`/`onSuccess`
콜백으로 편리하게 구현할 수 있다는 것을 알고 있었지만, 이번 과제는 기본적인 React
훅을 어떻게 활용하는지를 보여주는 데 취지가 있다고 판단해 라이브러리 사용을
최소화했습니다. 다만 가상화는 처음 접해본 전략이었고, 기본 훅만으로 직접
구현하기에는 현재 제 실력으로 한계가 있다고 판단해, 이 부분에 한해
라이브러리(`@tanstack/react-virtual`)를 도입했습니다.
 
## 1. 상태 구조
 
- 서버 상태와 클라이언트(낙관적) 상태를 어떻게 나눴나요? 어디서 관리하나요?
  - 별도로 분리하지 않았습니다. `Board.tsx`의 `tasks`(`useState<Task[]>`) 배열 하나에서
    서버 확정값과 낙관적 반영값을 함께 관리합니다. 다만 "지금 이 카드가 서버 확정을
    기다리는 중인가"만 `inFlight`(`useRef<Set<string>>`)로 별도 추적합니다.
- 이렇게 잡은 이유와, 고려했다가 버린 대안은?
  - **이유:** 이동/생성/수정/삭제가 전부 "낙관적 반영 → 요청 → 성공 시 교체 / 실패 시
    롤백"이라는 동일한 패턴입니다. 상태를 서버/클라이언트로 분리하면 오히려 두 상태를
    동기화하는 코드가 추가로 필요해져, 단일 배열로 관리하는 편이 더 단순하다고
    판단했습니다.
  - **훅 선택 기준:**
    - `useState`: 값이 바뀔 때 화면이 다시 그려져야 하는 것 (tasks, loading, isError,
      showForm, editing)
    - `useRef`: 로직 판단에만 쓰이고 화면과 무관한 것 (inFlight — 요청 중 여부)
    - `useEffect`: 마운트 시 1회성 동작 (초기 로드)
  - **일반 변수 대신 useRef를 쓴 이유:** 일반 변수(let)는 리렌더 시 컴포넌트 함수가
    처음부터 재실행되어 매번 초기화되므로 "요청 중" 정보가 유지되지 않습니다.
    `useState`는 값은 유지되지만 변경 시 불필요한 리렌더를 유발합니다(5,000개 카드가
    리렌더될 수 있음). `useRef`는 리렌더 간 값이 유지되면서 리렌더를 일으키지 않아
    이 용도에 적합했습니다.
  - **버린 대안 — TanStack Query:** 편의는 크지만, 과제 취지에 맞게 React가 제공하는
    훅들을 최대한 활용해 직접 구현하기 위해 채택하지 않았습니다.
## 2. 낙관적 업데이트 & 롤백
 
- 이동/수정/삭제/생성 시 UI를 먼저 반영하고 실패하면 어떻게 되돌리나요?
  - 네 가지 조작 모두 "화면 먼저 반영 → 서버 요청 → 성공 시 서버값 반영 / 실패 시
    되돌림" 흐름입니다.
    - **이동(moveTask):** status를 즉시 변경(`patchTask`) → 실패 시 이동 전 카드로
      복원(`rollbackTask`)
    - **수정(editTask):** 변경값을 즉시 반영(`patchTask`) → 실패 시 수정 전 카드로
      복원(`rollbackTask`)
    - **삭제(removeTask):** 목록에서 즉시 제거(`removeById`) → 실패 시 배열 끝에
      재삽입
    - **생성(addTask):** 임시 id(`temp-${Date.now()}`)로 카드를 즉시 추가 → 성공 시
      서버가 준 진짜 카드로 교체, 실패 시 임시 카드 제거(`removeById`) 
      
- 롤백을 "서버 재조회" vs "이전 스냅샷 복원" 중 무엇으로 했고 그 이유는?
  - **이전 스냅샷 복원**을 선택했습니다. 데이터가 5,000개라 실패 하나마다
    `getTasks()`로 전체를 다시 받아오면 비용이 크고(200~800ms 지연 + 5,000개
    재파싱), 그 사이 화면이 다시 로딩 상태로 보이거나 멈춰 보일 수 있어 낙관적
    업데이트의 취지와도 맞지 않습니다. 실패 이전에 붙잡아둔 해당 카드만 되돌리면
    즉시 복원되고 비용도 적습니다.
  - 전체 배열을 통째로 스냅샷 복원하지 않고 **해당 카드만** 되돌리는 이유는, 그
    사이 다른 카드에 대해 성공적으로 반영된 변경이 함께 사라지는 것을 막기
    위해서입니다.

## 3. 경쟁 상태(race condition)
- 같은 카드를 빠르게 연속 이동할 때 최종 상태가 어긋나지 않도록 무엇을 했나요?
  (요청 취소 / 버전·순번 비교 / last-write-wins 등)

**경쟁 상태 제어 방식 및 사유**
- **카드별 요청 직렬화:** `inFlight`(Set)에 요청 중인 카드 id를 저장하여, 요청 중이면 새 요청을 무시하고 요청이 끝나면(finally) 해제하는 방식을 택했습니다.
- **효과:** 한 카드당 항상 요청이 1개만 진행되므로 응답 순서가 뒤섞이거나 예전 version으로 인한 충돌(409)을 방지합니다.
- **디바운스+클린업 방식 대신 직렬화를 선택한 이유:** 
  - (200ms 지연 후 전송, 재이동 시 clearTimeout) 방식도 직접 구현해 보았으나, 200ms 지연 시간 동안 새로고침이나 탭을 닫아버리면 예약된 요청 자체가 브라우저 메모리에서 유실되어 서버에 전혀 반영되지 않는 치명적인 문제가 있었습니다.
  - 반면, 즉시 전송하는 직렬화 방식은 이 유실 창(Window)을 최소화할 수 있어 최종 확정했습니다.
- **검토 후 미채택(abort):** AbortController는 조회(GET)에는 적합하나 쓰기(PATCH)에는 부적합했습니다. 서버는 이미 처리했는데 클라이언트만 응답 대기를 취소해버리면 version이 꼬이기 때문입니다.

## 4. 대량 데이터 성능 (5,000개)
- 검색/필터/드래그를 부드럽게 만들기 위해 무엇을 선택했고 **무엇을 포기**했나요?
- 가능하면 측정 근거(예: 최적화 전/후 체감·프로파일)를 남겨 주세요.

**성능 최적화 선택과 포기한 것**
- **가상화(Virtualization) 선택:** `@tanstack/react-virtual` 라이브러리를 도입해 컬럼별 리스트를 가상화했습니다. 화면에 보이는 카드와 overscan 영역만 실제 DOM에 렌더링합니다.
- **가변 높이 대응:** `measureElement`로 실제 높이를 측정하여 카드 간 간격이 어긋나는 문제를 해결했습니다.
- **드래그 최적화:** 드래그 앤 드롭 타깃을 개별 카드가 아닌 컬럼(`section`) 전체로 잡아, 화면 밖 카드가 DOM에서 사라져도 드롭 동작에 지장이 없도록 했습니다.
- **포기한 것:** 카드 순서 유지 및 재배치는 다루지 않고 상태(status) 변경만 구현했습니다. 검색 및 필터 기능 자체는 시간 제약상 미구현했습니다.

**측정 근거**
- **최적화 전:** `document.querySelectorAll('.card').length`로 확인 시 실제 렌더링된 카드가 5,000개에 육박해 스크롤 시 극심한 버벅임이 체감되었습니다.
- **최적화 후:** 동일한 방법으로 확인 시 DOM에 40~60개 수준의 카드만 유지되어 스크롤 및 드래그가 매우 부드러워졌습니다.

## 5. 정답이 없던 결정들 (명세서에 명시되지 않은 항목)
- **네트워크 완전 단절** 시 동작을 어떻게 정의했나요? 왜?
- **409 충돌** 시 사용자에게 무엇을 보여주고 어떻게 해소하나요? 왜?
- 실패한 쓰기의 **재시도**: 자동 vs 수동 — 무엇을 택했고 이유는?

**네트워크 완전 단절**
- 별도의 분기 처리 없이 일반적인 쓰기 실패(500 에러 등)와 동일하게 처리하여 롤백 + alert 알림을 띄웁니다.
- **이유:** 클라이언트 입장에서 완전 단절과 서버 내부 오류를 완벽히 구분할 근거가 부족하며, 구분하더라도 사용자에게 제공할 수 있는 대응(다시 시도 유도)이 똑같기 때문입니다.

**409 충돌**
- 롤백이 아닌 **'화해(부분 갱신)'**로 처리합니다. 서버가 내려준 최신 상태(`payload.current`)로 해당 카드를 덮어쓰고, *"다른 곳에서 먼저 변경되었습니다. 최신 상태로 맞췄어요"* 라는 alert를 띄웁니다.
- **이유:** 409는 내 요청이 틀린 것이 아니라 그 사이 서버가 먼저 바뀐 것이므로, 내 낙관적 변경을 완전히 버리는 것보다 서버 최신값을 반영하는 것이 더 정확합니다. 전체 재조회를 하지 않은 이유는 5,000개 데이터의 재조회 비용을 회피하기 위해서입니다.

**실패한 쓰기의 재시도**
- 자동 재전송 대신 **'수동(사용자가 다시 시도)'**을 택했습니다.
- **이유:** 자동 재시도(`fetchWithRetry` 최대 3회)를 구현해 보았으나, 사용자가 이미 다른 조작을 이어가는 중에 뒤늦게 과거의 재시도가 뜬금없이 반영되면 오히려 극심한 UX 혼란을 초래한다고 판단했습니다. (초기 전체 로드 실패 시에만 수동 '다시 시도' 버튼 제공)

## 6. 그 외 트레이드오프 / 남은 한계
- 시간 제약으로 포기한 것, 알고 있는 버그·한계, 더 있었다면 했을 것.

**트레이드오프 및 남은 한계**
- **알림 방식:** 현재 단순 `alert`를 사용하고 있어 맥락이 다소 끊깁니다. `Toast` 알림으로 교체하면 더 매끄러운 UX가 가능하지만 우선순위를 고려해 미적용했습니다.
- **미구현 항목:** 앞서 언급한 바와 같이 검색, 필터, 그리고 다중 탭 동기화 기능은 시간 관계상 구현하지 못했습니다.
