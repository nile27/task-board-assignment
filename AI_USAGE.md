# AI 도구 활용 (AI_USAGE)

> AI가 준 답을 어떻게 검증·판단했는지를 정리합니다.

## 1. 사용한 도구
- Claude (설계 논의, 문제 인식 정리, 코드 작성/리뷰)
- Claude Code (코드베이스 분석, 기능별 커밋 분리, 반복 코드 적용)

## 2. 주로 어디에 썼나

| 작업 | 사용한 도구 | 활용 방식 |
|------|-------------|-----------|
| 문제 인식 · 요구사항 분해 | Claude | 명세를 문제 단위로 쪼개 논의 후 직접 정리 |
| 낙관적 업데이트 · 롤백 구조 설계 | Claude | 접근법 논의 후 직접 이해하며 적용 |
| 경쟁상태 처리 방식 비교 | Claude | 직렬화/큐잉/abort 트레이드오프 논의 후 선택 |
| 가상화 적용 | Claude | react-virtual 코드 작성 후 갭/높이 이슈 수정 |
| 코드베이스 함수 분석 | Claude Code | 파일별 함수 역할 정리 |
| 기능별 커밋 분리 | Claude Code | 커밋 단위 나눠 반영 |

## 3. AI 답을 그대로 쓰지 않고 검증·거부·수정한 사례

### 사례 1 — 코드 분석 오류 정정
- **AI 사용:** 코드 에이전트에게 기존 함수 역할 분석 요청
- **문제:** 에이전트가 `moveTask` 내 낙관적 업데이트 및 롤백이 "구현되어 있음"으로 잘못 분석 (실제는 주석 내 TODO 상태)
- **발견:** 원본 파일과 직접 대조하여 주석을 실제 구현으로 착각한 점 확인
- **조치:** "미구현" 상태로 정정 후 코드 기준으로 파악 다시 수행 (잘못된 문제 인식 방지)

### 사례 2 — 409 충돌 처리 방식 수정
- **AI 제안:** 409 충돌 시 `getTasks()`로 전체 데이터 재동기화
- **문제:** 5,000개 데이터 환경에서 전체 재조회는 비용이 큼. 낙관적 업데이트 취지에 부합하지 않음
- **조치:** 명세 확인 후 409 응답에 포함된 `current`(최신 상태) 데이터 활용, 전체 조회 대신 해당 카드만 부분 갱신하도록 수정

### 사례 3 — abort 용도 분리
- **검토:** 경쟁상태 해법으로 abort(AbortController) 제안받음
- **문제:** abort는 조회(GET)에는 유효하나 쓰기(PATCH)에 사용 시 서버 처리 결과를 클라이언트에서만 무시하게 되어 version 꼬임 발생
- **조치:** 쓰기 경쟁상태는 '요청 직렬화'로 해결, abort는 조회 재시도에만 적용하도록 용도 명확히 분리

### 사례 4 — 타 블로그 전략의 선별적 수용
- **검토:** 동시 낙관적 업데이트 관련 전략(React Query 계열) 참고
- **문제:** 해당 전략은 version 검증이 없고 refetch 비용이 낮음을 전제. 본 과제는 엄격한 version 검증(409)과 높은 refetch 비용(5,000개 데이터) 존재
- **조치:** 해당 전략 수용 거부. "응답 version 반영 + 직렬화 + 409 화해"라는 과제 맞춤형 로직으로 재구성

## 4. 스스로 판단해서 결정한 부분
- **경쟁상태 = 직렬화:** 큐잉 방식의 UX 이점을 인지하나 version 동기화 및 유지보수 복잡도 증가를 고려, 단순하고 명확한 직렬화 방식 채택 (DECISIONS 3 참고)

- **경쟁상태 자체 해결 (디바운스 + 클린업):** 직렬화(튕김 발생)와 큐잉(복잡성 증가)의 단점 보완을 위해 직접 설계. 화면은 즉시 반영하고 전송 단계에만 클린업 기반 디바운스 적용. 최종 이동 건만 전송해 정합성 확보 및 튕김 현상 제거 (DECISIONS 3 참고)
